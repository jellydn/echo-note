## Codebase Patterns
- Use Tauri v2 with React + TypeScript frontend
- Rust backend with tauri::command for exposing functions to frontend
- TypeScript strict mode enabled - must provide explicit types for buttons (type="button")
- CSS uses CSS variables and prefers-color-scheme for dark mode support
- Use `cargo tauri dev` for development with hot reload
- Run `cargo check` for Rust typecheck and `tsc --noEmit` for TypeScript typecheck
- Use `cargo clippy` for Rust linting
- Project uses bun for package management (bun.lock present)
- **SQLx with SQLite**: Use `sqlx::query!` macro with `chrono` feature for DateTime support
- **Database initialization**: Run migrations in `setup()` hook, store pool in `AppState`
- **DateTime handling**: Use RFC 3339 strings for JSON serialization, chrono DateTime<Utc> for DB
- **API Response pattern**: Use generic `ApiResponse<T>` struct for consistent responses
- **Settings pattern**: UPSERT via `ON CONFLICT(key) DO UPDATE`, defaults via `INSERT OR IGNORE`
- **Foreign keys**: Use `ON DELETE CASCADE` for auto-cleanup of related records
- **Audio thread pattern**: cpal Stream is not Send/Sync - use dedicated audio thread with message passing
- **Dual-device recording**: Spawn separate threads for microphone and BlackHole, mix audio streams with sample rate normalization
- **BlackHole detection**: Use system_profiler SPAudioDataType to detect BlackHole virtual audio device
- **Audio mixing**: Mix multiple streams with attenuation (0.75 factor) to prevent clipping
- **Resampling**: Linear interpolation for matching different sample rates between devices
- **HTTP streaming downloads**: Use `reqwest::Client` with `bytes_stream()` and `futures_util::StreamExt` for large file downloads
- **Progress events**: Use `app_handle.emit("event-name", payload)` to send progress updates to frontend
- **Model management**: Store models in app data directory (`~Library/Application Support/echo-note/models/`) with Hugging Face URLs
- **LLM integration**: Use `reqwest` with `json` feature for POST requests to Ollama API
- **Summary parsing**: Parse structured LLM output by extracting sections between markers (e.g., "KEY POINTS:", "DECISIONS:")
- **LLM prompt engineering**: Use structured prompts with clear section markers for consistent output parsing
- **API fallback pattern**: Check `llm_provider` setting, fetch API config from settings, route to appropriate implementation
- **React hooks order**: Import hooks from React in alphabetical order (`useCallback`, `useEffect`, `useRef`, `useState`)
- **Tauri invoke pattern**: Use `invoke<ApiResponse<T>>("command_name", { params })` for type-safe backend calls
- **Modal UI pattern**: Use overlay div with `position: fixed` and `z-index` for modal dialogs
- **CSS animations**: Use `@keyframes` with `animation` property for pulsing/recording indicators
- **Timer display**: Use `font-variant-numeric: tabular-nums` for consistent-width numbers in timers
- **Biome lint config**: Use `files.includes` with negative patterns (e.g., `!**/target`) to exclude build artifacts
- **Tauri event listening**: Use `listen<T>("event-name", callback)` from `@tauri-apps/api/event` for backend events
- **Cleanup pattern**: Store unlisten function in ref, call it in useEffect cleanup to prevent memory leaks
- **Processing UI pattern**: Use stage-based state machine (idle → transcribing → complete → generating → done)
- **Progress bar**: CSS-based with percentage width and smooth transitions
